# Minecraft RAG AI 챗봇 개발 히스토리 (Part 2: 대화 문맥 인지 및 스트리밍 도입)

## 1. 대화 기억 상실(Context Amnesia) 문제 해결
* **이슈:** 챗봇이 이전 대화의 주어(예: 워든)를 기억하지 못하고, 후속 질문("잡는 방법이 뭐야?")을 받았을 때 엉뚱한 문서(근거지 찾는 법 등)를 검색하여 대답하는 현상 발생.
* **해결 (기록 인지 검색기 도입):**
  * `langchain_classic.chains`의 `create_history_aware_retriever`를 도입.
  * **작동 원리:** 이전 대화 기록(`chat_history`)을 바탕으로 사용자의 질문을 독립적인 질문으로 재구성(Query Rewriting)한 뒤 Vector DB를 검색하도록 파이프라인(Chain) 개선.
  * 결과: "워든 잡는 방법 알려줘" ➡️ "그래서 체력은 몇이야?" 처럼 주어를 생략해도 문맥을 완벽하게 파악하고 답변 가능.

## 2. API Rate Limit (429 RESOURCE_EXHAUSTED) 이슈 및 최적화
* **이슈:** 질문을 연속으로 할 경우 `429 RESOURCE_EXHAUSTED` 에러 발생.
* **원인 분석:** 
  * '질문 재구성'과 '최종 답변 생성'을 위해 챗봇이 내부적으로 API를 2번씩 호출하게 됨.
  * 이로 인해 Google Gemini 무료 티어(Free Tier)의 분당 요청 제한(RPM, 보통 15~20회)에 2배 빠르게 도달함.
* **해결 (사용자의 예리한 피드백 반영 및 최적화):**
  * 처음에는 [이전 질문 + 현재 질문]을 병합하는 꼼수를 썼으나, **사용자의 지적대로 "주제가 완전히 전환될 때(예: 워든 -> 어부 주민)" 검색 품질이 심각하게 훼손되는 치명적인 결함** 발견.
  * **최종 해결책:** DB 검색(Retrieval)은 오직 **'현재 질문'**으로만 진행하여 주제 전환에 완벽하게 대응함. 대신, LLM이 최종 답변을 생성할 때 **'전체 대화 기록(chat_history)'**을 통째로 읽게 하여, 대명사 생략("체력은 몇이야?") 같은 문맥을 스스로 파악하고 대답하도록 설계.
  * 결과: **API 호출 횟수를 1회로 유지하면서도, 주제 전환과 문맥 유지 두 마리 토끼를 모두 잡는 데 성공.** 429 에러 발생 확률 대폭 감소.

## 3. 체감 속도 개선 (스트리밍 출력 도입)
* **이슈:** API 2회 호출로 인해 최종 답변이 화면에 한 번에 툭 튀어나오기까지의 대기 시간(지연)이 길어짐.
* **해결 (스트리밍 적용):**
  * `ChatGoogleGenerativeAI` 모델 설정에 `streaming=True` 추가.
  * 기존 `rag_chain.invoke()` 대신 `rag_chain.stream()` 메서드를 사용하여, 생성되는 답변 조각(chunk)을 실시간으로 받아옴.
  * Streamlit의 `st.empty().markdown()`을 활용하여 ChatGPT처럼 글자가 타자 치듯 실시간으로 출력되는 스트리밍(Streaming) 기능 적용.
  * 결과: 사용자가 답변을 기다리는 체감 대기 시간이 대폭 감소함.
